## 零、全自定义架构原则（核心设计约束）

> **本原则为最高优先级约束，所有模块必须遵循。**

### 0.1 设计动机

档案行业的国家标准（如 T22-2015）、地方标准、行业标准（DA/T 系列）以及各单位内部规范**均可能随时间调整**。例如：
- 归档章格子数量可能从 6 格变为 8 格或更多
- 单格尺寸可能调整
- 字段名称可能变化（"机构或问题"→"分类号"）
- 可能出现新的档案门类，需要全新的归档章样式

因此，程序中**不得硬编码任何归档章视觉参数**。所有值必须通过 JSON Schema 配置驱动。

### 0.2 全自定义参数清单

以下参数**全部可自定义**，无任何硬编码默认值（内置模板提供的仅是「初始预设」，用户可在 UI 中修改）：

#### ① 网格结构参数

| 参数 | 类型 | 说明 | T22-2015 预设 |
|------|------|------|--------------|
| `rows` | `usize` | 行数 | `2` |
| `cols` | `usize` | 列数 | `3` |
| `cell_size_mm` | `[f32; 2]` | 单个格子尺寸 [高mm, 宽mm] | `[3, 15]` |
| `cell_sizes_mm` | `[[f32; 2]]`（可选）| 每格独立尺寸（覆盖统一尺寸）| `null`（使用统一尺寸）|

#### ② 合并单元格（Merged Cells）

支持跨行/跨列合并，实现不规则布局：

```json
"merged_cells": [
  { "row": 0, "col": 0, "rowspan": 1, "colspan": 2, "label": "全宗号-年度组合" }
]
```

- `rowspan`：向下合并行数（`1` = 不合并）
- `colspan`：向右合并列数（`1` = 不合并）
- 被合并的区域内的其他 cell 定义应被忽略（渲染时跳过）

#### ③ 每个 Cell 的完整属性

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | `string` | ✅ | 唯一标识符，关联 field_registry |
| `label` | `string` | ✅ | 显示文字 |
| `row` | `usize` | ✅ | 所在行（0-based）|
| `col` | `usize` | ✅ | 所在列（0-based）|
| `source` | `string` | ✅ | 数据来源类型 |
| `required` | `bool` | ✅ | 是否必填 |
| `rowspan` | `usize` | ❌ | 向下合并几行（默认 1）|
| `colspan` | `usize` | ❌ | 向右合并几列（默认 1）|
| `text_align` | `string` | ❌ | 对齐方式：`left`/`center`/`right`（默认 `center`）|
| `font_size_pt` | `f32` | ❌ | 本格独立字号（覆盖全局 font.size_pt）|
| `value_format` | `string` | ❌ | 值格式化模板，如 `{value}页` |

#### ④ 整体外观参数

| 参数 | 类型 | 所属节点 | 说明 | T22-2015 预设 |
|------|------|---------|------|--------------|
| 锚点 | `string` | position.anchor | `top-right`/`top-left`/`bottom-right`/`bottom-left`/`custom` | `top-right` |
| 偏移量 | `[f32; 2]` | position.offset_mm | [X偏移mm, Y偏移mm] | `[5, 5]` |
| 适用页面 | `string` | position.page_scope | `first-only`/`all-pages`/`custom` | `first-only` |
| 字体族 | `string` | font.family | 字体名称 | `SimFang` |
| 字号 | `f32` | font.size_pt | 全局默认字号 | `9` |
| 备用字体 | `string` | font.fallback | 主字体不可用时的备用字体 | `SimSun` |
| 边框宽度 | `f32` | border.width_pt | 外边框+内分隔线宽度 | `0.5` |
| 边框颜色 | `[u8; 3]` | border.color | RGB 颜色 | `[0, 0, 0]` |
| 内边距 | `f32` | padding_mm | 格子内容距边框的距离（mm）| `0.5` |

#### ⑤ 页码参数（已在 §6 完整定义）

所有页码参数同样遵循全自定义原则（范围/格式/位置/外观/双面模式等），此处不重复列举。

### 0.3 架构约束规则

1. **零硬编码**：Rust 绘制代码和 React 渲染代码中，不得出现任何数值字面量（如 `3.0`、`15.0`、`6`、`45` 等）。所有数值必须来自 Schema。
2. **Schema 即真理**：JSON Schema 是唯一数据源。前端编辑器修改 Schema → 后端读取同一 Schema → 绘制结果一致。
3. **内置模板 = 预设值**：程序附带的 4 套内置模板（WS/KJ/KJ_ACCT/SX）仅为「开箱即用的起点」，用户可以基于它们创建任意变体。
4. **向前兼容**：新增可选字段时，旧 Schema 缺失则使用合理的 fallback 默认值（在 Rust 侧的 `Default` trait 或 `From` 实现中处理）。
5. **校验分离**：Schema 结构校验（如 row/col 不越界）由 Rust 后端执行；UI 层提供实时预览反馈。

## 一、需求梳理（最终确认版）

### 1.1 核心功能

| 功能 | 说明 |
|------|------|
| 批量扫描 | 递归遍历多层文件夹下全部 PDF 文件 |
| 归档章加盖 | **仅每件文件首页**右上角加印符合对应标准的归档章 |
| 页码添加 | **每页**都需要加页码，支持全量自定义（范围/格式/位置/外观）|
| 档案类型模板 | 支持多套规则并存（文书/科技/会计/声像等），自动匹配或手动切换 |
| 字段来源自适应 | 支持：固定值 / 从路径提取 / 从文件名解析 / 自动读取 |
| 文件夹结构自适应 | 支持多全宗号、多年度、多保管期限同期处理（批次隔离）|
| 空白检测（辅助）| 检测右上角是否有内容，**仅作提示**，不阻塞流程 |
| 手动选址 | 用户点击 PDF 页面图像指定归档章位置，坐标自动记忆复用 |
| 配置持久化 | 用户固化值、手动坐标、处理历史、配置方案均持久保存 |

### 1.2 归档章格式（依据 T22-2015 附录A）

> **T22-2015《归档文件整理规则》附录A 规范**：单格高 3mm × 宽 15mm，整体为 2行×3列矩形分格章。

```
┌─────────────┬─────────────┬─────────────┐
│  全宗号      │   年度       │   件号       │
├─────────────┼─────────────┼─────────────┤
│机构或问题*  │ 保管期限     │   页数       │
└─────────────┴─────────────┴─────────────┘
```

- **章整体尺寸**：高 6mm × 宽 45mm（2行×3列，单格 3mm×15mm）
- **PDF 坐标参考**：约 17pt（高）× 128pt（宽）
- **位置**：首页右上角，距右边距 5mm，距顶边距 5mm（可配置）
- **加盖范围**：仅首页
- **中文字体**：仿宋（SimFang），随应用打包或读取系统字体
- **注**：标有"*"号的"机构或问题"为选择项，可按需留空

> **注意**：不同档案类型的归档章格式可能不同（格子数量、字段名称），通过「档案类型模板」动态配置。

### 1.3 文件命名规律（文书档案示例）

```
根目录/
├── 0226/2021/D30/0226-GL，2021-D30-0001.pdf
├── 0226/2021/Y/   0226-WS，2021-Y-0005.pdf
├── 0360/2024/D10/...
```

**Regex 解析模式（Rust，兼容中文）：**
```
^(?<fonds>\d+)-(?<org>[\w\u4e00-\u9fff]+)[，,]?(?<year>\d{4})[，,-](?<retention>[A-Z]\d*)(?:-[A-Za-z\u4e00-\u9fff]+)?[，,-](?<docnum>\d+)\.(?i:pdf)$
```

| 捕获组 | 含义 | 示例 | 对应归档章字段 |
|--------|------|------|---------------|
| `fonds` | 全宗号 | `0226` | 全宗号（source=config 或 path）|
| `org` | 机构代码 | `GL`、`ZTJ`、`办公室` | 机构或问题（source=filename）|
| `year` | 年度 | `2021` | 年度（source=path 或 config）|
| `retention` | 保管期限代码 | `D30`、`Y` | 保管期限（source=config_or_path）|
| `docnum` | 件号 | `0005` | 件号（source=filename，显示时去前导零）|

**保管期限代码映射：**

| 代码 | 显示文字 |
|------|---------|
| `Y` | 永久 |
| `D30` | 30年 |
| `D10` | 10年 |
| `D3` | 3年 |

## 二、档案类型模板系统（核心架构）

> **设计目标**：程序不再是「一把钥匙开一把锁」，而是变成了一个「钥匙工厂」——支持多套规则并存，自动匹配生效。

### 2.1 核心概念：档案门类配置模板

程序不再只有一套全局设置，而是支持创建多个「档案类型模板」。每个模板独立包含：

| 模板私有配置 | 说明 |
|-------------|------|
| 专属文件夹映射规则 | 路径怎么解析 |
| 专属文件名正则规则 | 文件名怎么拆解 |
| 专属归档章样式 | 格子数量、字段名称（会计档案可能多一个「月份」字段）|
| 专属页码样式 | 不同门类的页码规范可能不同 |
| 字段注册表 | 该模板下所有可用字段的定义 |

### 2.2 前端交互设计

#### 主界面布局变更

```
┌─────────────────────────────────────────────────┐
│  工具栏：［档案类型 ▾］［保存模板］［管理模板］│  ← 新增：档案类型切换
├─────────────────────────────────────────────────┤
│                                                 │
│  A区：全局固化值（随模板切换自动更新）            │
│  B区：文件夹映射（随模板切换自动更新）            │
│  C区：文件名解析（随模板切换自动更新）            │
│                                                 │
├─────────────────────────────────────────────────┤
│  批次管理 + 文件预览表格                          │
└─────────────────────────────────────────────────┘
```

#### 步骤一：新建/选择模板

用户在界面上可以看到预设的模板，例如：**文书档案(WS)**、**科技档案(KJ)**、**会计档案(KJ_ACCT)**。用户可以点击「新建类型」，比如命名为「基建档案」。

#### 步骤二：为当前类型配置专属规则

当用户选中「科技档案」时，界面上的配置区（文件夹映射、文件名解析、归档章样式）会自动切换成该类型之前保存好的规则。

**场景演示：**
- 用户选"文书档案" → 文件夹映射显示为：`年度/保管期限/机构` → 文件名正则显示为：`全宗号-机构,年度-保管期限-件号.pdf`
- 用户切到"会计档案" → 文件夹映射自动变为：`年度/月份/凭证类型` → 文件名正则自动变为：`全宗号-凭证字号-顺序号.pdf`

#### 步骤三：智能推荐（置信度算法）

> **设计原则**：只作提示，不自动切换——用户可能把多种档案放在同一个根目录下。

**触发时机**：用户拖入根文件夹，程序扫描第一层文件夹名称和文件样本。

**置信度算法：**

```rust
/// 计算文件夹与某模板的匹配置信度（0~100）
/// 
/// 评分规则：
/// - 文件夹名含门类关键词（凭证/账簿/报表 → 会计；项目/工程/设计 → 科技）：+40%
/// - 文件名符合某模板的正则：+50%
/// - 文件夹层级数与某模板一致：+10%
pub fn calc_template_match_confidence(
    sample_paths: &[PathBuf],
    sample_filenames: &[String],
    template: &ArchiveTemplate,
) -> u8 {
    let mut score: u8 = 0;
    
    // 1. 关键词匹配（40分）
    let keyword_map = HashMap::from([
        ("WS",  vec!["文书", "WS"]),
        ("KJ",  vec!["科技", "项目", "工程", "设计"]),
        ("KJ_ACCT", vec!["会计", "凭证", "账簿", "报表"]),
        ("SX",  vec!["声像", "照片", "录音", "录像"]),
    ]);
    if let Some(keywords) = keyword_map.get(template.code.as_str()) {
        for path in sample_paths {
            let dir_name = path.file_name().unwrap_or_default().to_string_lossy();
            for kw in keywords {
                if dir_name.contains(kw) {
                    score = score.saturating_add(40);
                    break;
                }
            }
        }
    }
    
    // 2. 文件名正则匹配（50分）
    if let Ok(re) = Regex::new(&template.filename_regex_pattern) {
        for fname in sample_filenames {
            if re.is_match(fname) {
                score = score.saturating_add(50);
                break;
            }
        }
    }
    
    // 3. 文件夹层级数一致（10分）
    // （从 folder_mapping_json 中解析期望的层级数，与样本比较）
    
    score.min(100)
}
```

**交互流程：**
```
扫描完成 → 程序计算与各个模板的匹配置信度
         → 置信度 >= 80%：弹窗提示「检测到疑似【会计档案】，是否应用该模板？」
         → 置信度 < 80%：不弹窗，用户手动选择
```

### 2.3 字段注册表（Field Registry）概念

程序内部不再写死「机构」、「问题」这些词，而是通过**字段注册表**实现动态字段。

**字段注册表结构（JSON 示例）：**

```json
{
  "field_registry": [
    { "field_name": "fonds_code",  "label": "全宗号",   "default_source": "config",    "required": true  },
    { "field_name": "year",        "label": "年度",     "default_source": "path",       "required": true  },
    { "field_name": "doc_num",     "label": "件号",     "default_source": "filename",   "required": true  },
    { "field_name": "org",         "label": "机构或问题", "default_source": "filename", "required": false },
    { "field_name": "retention",   "label": "保管期限",  "default_source": "config_or_path", "required": true },
    { "field_name": "page_count",  "label": "页数",     "default_source": "pdf_metadata", "required": true }
  ]
}
```

**配置科技档案时**，用户把正则的第二个分组命名为 `project_code`（项目代号），并在字段注册表中添加：
```json
{ "field_name": "project_code", "label": "项目代号", "default_source": "filename", "required": true }
```

**动态渲染**：在配置「归档章样式」时，表格里的单元格数据源下拉框中，会动态列出当前模板下所有可用的字段（如 `org`, `project_code`, `year` 等）。用户只需把「项目代号」拖到归档章的第二个格子里即可。

### 2.4 归档章可视化编辑器

> 不同档案类型的归档章格子数量不同（文书 3×2=6 格，科技可能 4×2=8 格）。

**前端需支持：**
- 「增加/删除格子」按钮
- 「拖拽字段到格子」交互（从字段注册表拖入）
- 格子支持合并（如某些档案类型的归档章有跨列大格子）
- 实时预览归档章效果

## 三、前端三区配置设计（核心交互）

> 设计原则：**所见即所得的配置 + 灵活的变量映射**。用户不需写代码，通过可视化操作完成所有配置。  
> 以下三区配置均**跟随当前选中的档案类型模板**自动切换。

### 3.1 A 区：全局固化值配置（手动输入区）

此区域对应您在需求中提出的「固化值」，通常由档案员在加工前统一设定。

| 字段 | 控件类型 | 说明 |
|------|---------|------|
| 全宗号 | 输入框 + 历史下拉 | 支持保存历史常用全宗号，下拉选择 |
| 默认保管期限 | 下拉选择 | 永久 / 30年 / 10年 / 3年 |
| 默认年度 | 数字输入框 | 默认读取当前系统年份，允许修改 |
| 归档章位置偏移 | X/Y mm 输入 | 距离右上角的偏移量 |

**设计逻辑**：这些值作为「兜底数据」——如果后续无法从文件名或文件夹中解析，就使用这里的值。

**配置方案保存/加载**：
- 「保存当前配置为方案」→ 命名保存（如「2024文书档案加工」）
- 「加载方案」→ 下拉选择，自动填充 A/B/C 三区所有配置
- 方案存储为 JSON 文件，便于迁移和复用

### 3.2 B 区：文件夹结构映射（动态解析区）

针对「完整的文件夹结构」，采用**「路径变量映射」**功能。

**界面设计**：
1. 用户提供「示例路径」（如 `D:\档案加工\2024年\文书档案\永久\办公室\001号.pdf`）
2. 程序自动将路径拆分为层级，每一级后面提供一个下拉框，让用户选择该层级的含义

**映射示例**：

| 路径层级 | 用户映射选择 | 对应变量 |
|---------|-------------|---------|
| `2024年` | → `{年度}` | `path_level: 0` |
| `文书档案` | → 忽略 | — |
| `永久` | → `{保管期限}` | `path_level: 1` |
| `办公室` | → `{机构或问题}` | `path_level: 2` |

**支持的映射类型**：`年度` / `全宗号` / `保管期限` / `机构或问题` / `忽略此级`

### 3.3 C 区：文件名档号解析（正则可视化区）

针对「完整的文件名档号信息」，提供**「正则捕获组命名」工具**。

**界面设计**：

1. **示例文件名输入**：用户输入一个文件名（如 `2024-WS-BGS-0005.pdf`）
2. **规则编写**：提供一个预设模板（如 `年度-门类-机构-件号`），用户点击「生成正则」，程序自动生成命名捕获组正则
3. **实时预览**：下方实时显示解析结果，例如：`年度: 2024 | 机构: BGS | 件号: 0005`
4. **字段自定义**：用户可以随意增加或减少捕获组（例如增加一个「问题代码」）

**正则生成规则**：
- 数字段 → `(?<name>\d+)`
- 英文/字母段 → `(?<name>[A-Za-z]+)`
- 中文段 → `(?<name>[\u4e00-\u9fff]+)`
- 混合段 → `(?<name>[\w\u4e00-\u9fff]+)`

> **重要**：C 区不是「全自动生成」，而是「生成后允许用户手动编辑正则」，并始终显示实时解析预览。

### 3.4 配置区交互流程图

```
用户选择档案类型模板（或程序自动推荐）
                    ↓
用户粘贴示例路径 → B区自动拆分层级 → 用户下拉映射每级含义
                    ↓
用户粘贴示例文件名 → C区自动生成正则 → 用户微调正则 → 实时预览解析结果
                    ↓
                A区填入兜底值（全宗号/年度/保管期限）
                    ↓
                「保存为模板」→ 下次直接加载
```

## 四、字段来源三层架构设计

### 4.1 设计背景

用户同期可能处理**多个全宗号、多个年度、多个保管期限**的文件。
全局固化值（`config`）只能覆盖部分场景，需要更灵活的设计。

### 4.2 每个字段独立配置来源

| 来源类型 | 值 | 适用场景 |
|---------|------|---------|
| `config` | A 区界面全局输入值 | 同期只处理一个全宗号/年度 |
| `path` | B 区文件夹路径对应层级自动提取 | 文件夹结构规范（如 `全宗号/年度/保管期限/`）|
| `filename` | C 区文件名 Regex 命名捕获组提取 | 件号、机构代码等嵌在文件名中 |
| `pdf_metadata` | PDF 实际页数 | 页数（无需配置）|
| `config_or_path` | 优先 config，空则取 path | 保管期限（多数固定，少数不同）|

### 4.3 文件夹结构自动检测（辅助功能）

程序扫描根目录后，自动分析文件夹名称规律，**作为 B 区映射的默认建议**：

```
检测逻辑：
  第1级文件夹全为数字（4-6位）→ 默认映射为「全宗号」
  第2级文件夹全为4位数字    → 默认映射为「年度」
  第3级文件夹含"D"或"Y"    → 默认映射为「保管期限」
```

用户可在 B 区界面上调整映射关系，调整后随档案类型模板保存。

## 五、批次隔离架构（多全宗多年度并发）

> 如果用户同时拖入 `全宗A/2023` 和 `全宗B/2024` 的文件夹，程序不能只用一套全局配置。

### 5.1 设计思路：基于「任务批次」的配置隔离

**扫描阶段**：
- 用户选择根目录进行扫描
- 程序根据 B 区「文件夹映射规则」，自动将文件划分为不同的「逻辑批次」
- 每个批次有独立的上下文（全宗号、年度、保管期限默认值）

**处理阶段**：
- 处理 `全宗A` 的文件时，程序自动调用 `全宗A` 对应的全宗号、年度配置
- 处理 `全宗B` 的文件时，自动切换上下文

### 5.2 前端批次管理界面

扫描完成后，展示「发现的批次列表」：

| 批次名称 | 全宗号 | 年度 | 保管期限 | 文件数 | 状态 |
|---------|--------|------|---------|--------|------|
| 批次#1 | 0226 | 2021 | D30 | 156 | 待处理 |
| 批次#2 | 0226 | 2021 | Y | 89 | 待处理 |
| 批次#3 | 0360 | 2024 | D10 | 312 | 待处理 |

- 支持「全选/反选」批次
- 支持「批次单独重试」：某批次出错可单独重跑
- 支持「批次配置覆盖」：选中批次后可单独修改该批次的固化值

## 六、页码全量自定义设计

### 6.1 自定义维度完整清单

#### ① 适用范围

| 选项 | 说明 |
|------|------|
| 全部文件 | 批次中所有 PDF 均加页码 |
| 仅已盖章文件 | 只给成功加盖归档章的 PDF 加页码 |
| 指定文件 | 在预览表格中勾选目标文件 |

#### ② 添加页面

| 选项 | 说明 |
|------|------|
| 全部页面 | 第1页到最后一页均加 |
| 跳过首页 | 首页（封面）不加页码（归档章已含页数信息）|
| 仅奇数页 / 仅偶数页 | 按需选择 |
| 指定范围 | 手动指定（如：`3-10, 15-20`）|

#### ③ 编号规则

| 选项 | 说明 |
|------|------|
| 逐件重排 | 每个文件独立编号，均从第 N 页开始（档案规范默认）|
| 连续编号 | 跨文件连续编号（如文件1有3页，文件2从第4页开始）|
| 起始号 | 自定义起始数字（续编场景）|
| 显示格式 | `{current}` / `第{current}页` / `{current}/{total}` / 完全自定义 |
| 补零 | `001`（3位补零）/ `0001`（4位补零）/ 不补零 |

#### ④ 位置与布局（完整版）

| 选项 | 说明 |
|------|------|
| 垂直位置 | 页眉（上）/ 页脚（下）|
| 水平位置 | 左侧 / 居中 / 右侧|
| 双面模式（奇偶页不同）| **奇数页：右侧（外侧）/ 偶数页：左侧（外侧）** |
| 双面模式（内侧）| 奇数页：左侧（内侧）/ 偶数页：右侧（内侧）|
| 边距微调 | X轴（左右）和 Y轴（上下）的毫米级偏移量输入框 |
| 距版心距离 | 可选「紧贴版心」或「固定距页边」|

> **档案行业规范提示**：双面打印时，页码通常在「外侧」（奇数页右、偶数页左）。

#### ⑤ 字体与外观

| 选项 | 说明 |
|------|------|
| 字体 | 支持调用系统字体（宋体、黑体、Times New Roman 等）|
| 字号 | 自定义磅数（如 10.5pt / 五号）|
| 颜色 | 颜色选择器（通常档案页码要求为黑色，但需提供自定义能力）|

| 透明度 | 0-100% 调节（防止遮挡文字时影响阅读）|

### 6.2 页码与归档章的位置关系

- 归档章位置：可独立配置（默认右上角）
- 页码位置：可独立配置（默认页脚居中，或右上角归档章右侧）
- 建议默认：归档章在首页右上角，页码在所有页面页脚居中

## 八、数据库设计（SQLite）

### 8.1 为什么需要数据库？

经过讨论，确认需要 SQLite 用于：

1. **去重**：记录已处理文件，避免重复盖章
2. **记住手动坐标**：用户手动选址后保存，同批其他文件可复用
3. **断点续传**：大批量处理中断后，重启可跳过已完成文件
4. **批次隔离**：多全宗多年度并发时，每个批次有独立上下文
5. **模板管理**：存储多套档案类型模板，支持内置+自定义

### 8.2 表结构

```sql
-- 档案类型模板表（核心新增）
CREATE TABLE IF NOT EXISTS archive_templates (
    id                 INTEGER PRIMARY KEY,
    code               TEXT NOT NULL UNIQUE,   -- 'WS' / 'KJ' / 'KJ_ACCT' / 'SX' / 用户自定义
    name               TEXT NOT NULL,          -- '文书档案' / '科技档案' / '会计档案' / '声像档案'
    description        TEXT,
    is_builtin         INTEGER DEFAULT 0,      -- 0=自定义(可编辑/删除), 1=内置(不可删除)
    folder_mapping     TEXT NOT NULL,           -- JSON: 文件夹层级→字段映射
    filename_regex     TEXT NOT NULL,           -- JSON: 正则 + 捕获组→字段映射
    field_registry     TEXT NOT NULL,           -- JSON: 该模板所有可用字段的注册表
    stamp_schema       TEXT NOT NULL,           -- JSON: 归档章结构（cells 数组 + field_registry）
    page_number_config TEXT NOT NULL,           -- JSON: 页码默认配置
    created_at         DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at         DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 配置方案表（A/B/C 区完整配置快照，可选功能）
CREATE TABLE IF NOT EXISTS config_schemes (
    id          INTEGER PRIMARY KEY,
    name        TEXT NOT NULL UNIQUE,
    template_id INTEGER REFERENCES archive_templates(id) ON DELETE SET NULL,
    scheme_json TEXT NOT NULL,            -- 完整 A/B/C 区配置 JSON 快照
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 批次表（多全宗多年度隔离）
CREATE TABLE IF NOT EXISTS batches (
    id               INTEGER PRIMARY KEY,
    project_id       INTEGER REFERENCES projects(id) ON DELETE CASCADE,
    template_id      INTEGER REFERENCES archive_templates(id),
    batch_name       TEXT NOT NULL,       -- 如 "0226-2021-D30"
    fonds_code       TEXT,                -- 本批次全宗号（覆盖全局配置）
    year             TEXT,                -- 本批次年度
    retention        TEXT,                -- 本批次保管期限
    file_count       INTEGER DEFAULT 0,
    status           TEXT DEFAULT 'pending',  -- pending | processing | completed | failed
    created_at       DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 项目表
CREATE TABLE IF NOT EXISTS projects (
    id          INTEGER PRIMARY KEY,
    name        TEXT NOT NULL,
    root_path   TEXT NOT NULL,
    schema_json TEXT NOT NULL,
    config_json TEXT NOT NULL,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 文件表
CREATE TABLE IF NOT EXISTS files (
    id               INTEGER PRIMARY KEY,
    batch_id         INTEGER REFERENCES batches(id) ON DELETE CASCADE,
    project_id       INTEGER REFERENCES projects(id) ON DELETE CASCADE,
    template_id      INTEGER REFERENCES archive_templates(id),
    file_path        TEXT NOT NULL,        -- 相对于根目录的路径
    abs_path         TEXT NOT NULL UNIQUE, -- 绝对路径（去重关键）
    file_mtime       INTEGER NOT NULL,     -- 文件修改时间（判断是否需要重盖）
    status           TEXT DEFAULT 'pending',
    -- status: pending | processing | manual_needed | completed | failed
    error_msg        TEXT,
    extracted_json   TEXT,                -- 解析出的字段 JSON
    manual_x_mm      REAL,                -- 手动盖章 X 坐标（mm）
    manual_y_mm      REAL,                -- 手动盖章 Y 坐标（mm）
    schema_version   TEXT,                -- 盖章时使用的 schema 版本
    stamped_at       DATETIME,            -- 盖章完成时间
    created_at       DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at       DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_files_abs_path  ON files(abs_path);
CREATE INDEX IF NOT EXISTS idx_files_status    ON files(status);
CREATE INDEX IF NOT EXISTS idx_files_batch     ON files(batch_id);
CREATE INDEX IF NOT EXISTS idx_files_template  ON files(template_id);
CREATE INDEX IF NOT EXISTS idx_files_project   ON files(project_id);
```

## 十、PDF 处理方案

### 10.0 关键要求：保留双层 PDF 结构

> **档案行业规范**：最终交付必须是双层PDF（图像层+OCR文本层），盖章和加页码不能破坏此结构。

**技术挑战**：
- 双层PDF包含：底层扫描图像 + 上层OCR文本（不可见但可选中）
- 错误的处理方式：渲染页面为图像→添加内容→写回（会丢失文本层）
- 正确的处理方式：在保留原有内容的基础上，追加新的内容或注释

**解决方案**：
1. **使用PDF注释（Annotations）**：将归档章和页码作为注释添加到页面
2. **追加内容流（Content Stream）**：在原有内容流之后追加新的绘制指令
3. **避免使用图像渲染方式**：不将PDF页面渲染为图像再写回

### 10.1 技术选型：纯 Rust（优先）

**方案：**

| 库 | 用途 | 双层PDF处理策略 |
|---|---------|--------------|
| `pdfium-render` | PDF 渲染（仅用于预览和空白检测辅助） | 不参与最终输出 |
| `lopdf` | PDF 编辑（追加注释或内容流）| **保留原有内容，追加新内容** |

**关键策略**：
- **归档章**：作为 `Stamp` 注释或 `FreeText` 注释添加到首页
- **页码**：作为 `FreeText` 注释添加到每页
- **优点**：注释是PDF标准功能，不会破坏原有内容流和文本层
- **缺点**：注释可能被用户隐藏（但打印时默认会显示）

**备选策略（更彻底）**：
- 使用 `lopdf` 读取原页面内容流
- 创建新的内容流，包含归档章/页码的绘制指令
- 将新内容流追加到页面（PDF支持多内容流拼接）
- 这样文本层完全保留，归档章和页码成为页面内容的永久部分

**中文字体处理：**

```rust
/// 检查并加载中文字体
pub fn ensure_chinese_font(assets_dir: &Path) -> Result<Vec<u8>> {
    let candidates = [
        "C:/Windows/Fonts/simfang.ttf",
        "C:/Windows/Fonts/simsun.ttc",
        &format!("{}/fonts/simfang.ttf", assets_dir.display()),
    ];
    for path in candidates {
        if Path::new(path).exists() {
            return Ok(std::fs::read(path)?);
        }
    }
    Err("未找到中文字体，请在设置中指定字体文件路径".into())
}

/// 使用 lopdf 追加内容流（保留双层PDF结构）
/// 
/// 策略：读取原页面内容流，创建新内容流追加归档章/页码绘制指令
/// 不替换原内容，而是追加，从而保留图像层和文本层
pub fn add_stamp_preserve_layers(
    pdf_bytes: &[u8],
    stamp_schema: &StampSchema,
    page_number_config: &PageNumberConfig,
) -> Result<Vec<u8>, String> {
    let mut doc = lopdf::Document::load_mem(pdf_bytes)
        .map_err(|e| format!("加载PDF失败: {}", e))?;
    
    // 获取所有页面
    let pages = doc.get_pages();
    
    for (page_num, page_id) in pages.iter() {
        // 获取原页面内容流（不删除！）
        let original_content = doc.get_object(*page_id);
        
        // 创建新的内容流，包含归档章/页码绘制指令
        let new_content_stream = generate_stamp_content_stream(stamp_schema, page_number_config, *page_num)?;
        
        // 将新内容流追加到页面（PDF规范支持多内容流）
        // 这样原图像层和文本层完全保留
        append_content_stream(&mut doc, *page_id, new_content_stream)?;
    }
    
    // 保存为新的PDF（保留所有原始内容）
    let mut output = Vec::new();
    doc.save_to(&mut output)
        .map_err(|e| format!("保存PDF失败: {}", e))?;
    
    Ok(output)
}

/// 生成归档章和页码的内容流（PDF绘制指令）
fn generate_stamp_content_stream(
    schema: &StampSchema,
    page_config: &PageNumberConfig,
    page_num: u32,
) -> Result<Vec<u8>, String> {
    let mut stream = Vec::new();
    
    // 1. 设置字体和字号
    writeln!(stream, "BT").unwrap();  // Begin Text
    writeln!(stream, "/F1 9 Tf").unwrap();  // 使用字体F1，9pt
    
    // 2. 绘制归档章（仅在首页）
    if page_num == 1 {
        draw_stamp_grid(&mut stream, schema)?;
    }
    
    // 3. 绘制页码（根据配置）
    if should_add_page_number(page_num, page_config) {
        draw_page_number(&mut stream, page_num, page_config)?;
    }
    
    writeln!(stream, "ET").unwrap();  // End Text
    
    Ok(stream)
}
```

**如果纯 Rust 方案在中文字体嵌入上卡住，降级方案：**
- 用 Python Sidecar（`reportlab` 生成章图层 + `pypdf` 叠加）
- **关键**：使用 `pypdf` 的 `add_transformation` 或 `merge_transformed_page` 保留原PDF结构
- 通信协议：stdin 传入 JSON，stdout 逐行输出进度
- 打包：`pyinstaller --onefile stamp_pdf.py`，输出 `stamp_pdf.exe` 放入 `src-tauri/sidecar/`

### 10.2 归档章坐标计算

```rust
/// 计算归档章左上角坐标（PDF 坐标系：原点在左下角）
///
/// # 参数
/// - page_w, page_h: PDF 页面尺寸（pt）
/// - anchor: 锚点位置（top-right / top-left / custom）
/// - offset_mm: [X偏移, Y偏移]（mm）
/// - stamp_w, stamp_h: 归档章尺寸（pt）
pub fn calc_stamp_origin(
    page_w: f32, page_h: f32,
    anchor: &Anchor,
    offset_mm: &[f32; 2],
    stamp_w: f32, stamp_h: f32,
) -> (f32, f32) {
    let mm_to_pt = 2.835;
    let (ox, oy) = (offset_mm[0] * mm_to_pt, offset_mm[1] * mm_to_pt);
    match anchor {
        Anchor::TopRight => {
            // 右上角：x = 页宽 - X偏移 - 章宽；y = 页高 - Y偏移 - 章高
            (page_w - ox - stamp_w, page_h - oy - stamp_h)
        }
        Anchor::TopLeft => (ox, page_h - oy - stamp_h),
        Anchor::Custom => (ox, oy),
    }
}
```

### 10.3 归档章加盖策略（核心流程）

**不再以空白检测为主，改为「默认坐标 + 冲突检测 + 手动兜底」：**

```
处理每件文件：
  1. 按默认坐标（右上角，距顶 15mm，距右 10mm）计算盖章位置
  2. 渲染首页为图像（仅用于检测，不用于输出），检测归档章区域像素（冲突检测）
     - 如果检测到文字重叠（非白像素 > 10%）→ 弹窗让用户手动点选
     - 如果用户手动点选 → 记录坐标到 SQLite，同类文件复用
     - 如果用户跳过 → 仍按默认坐标盖章（用户确认可以覆盖）
  3. 使用 lopdf 追加内容流（不替换原内容）
     - 读取原PDF字节
     - 创建新的内容流，包含归档章绘制指令
     - 将新内容流追加到首页
     - **保留原图像层和文本层**
  4. 遍历所有页面，按页码配置追加页码内容流
  5. 保存到 <原目录>/_stamped/ 目录
  6. 标记 SQLite 中该文件 status = 'completed'
```

**关键原则**：
- **绝不渲染页面为图像再写回**（会丢失文本层）
- **使用内容流追加**（PDF规范支持多内容流拼接）
- **保留所有原始内容**（图像层、文本层、元数据等）

### 10.4 空白检测（辅助功能，可选启用）

```rust
/// 检测 PDF 第一页指定区域是否空白
/// 返回：(是否空白, 非白像素比例 0.0~1.0)
/// 
/// ⚠️ 此函数仅作「辅助提示」，不参与核心流程判断
pub fn detect_blank_region(
    img: &DynamicImage,
    page_w_pt: f32, page_h_pt: f32,
    schema: &StampSchema,
) -> (bool, f32) {
    let mm_to_pt = 2.835;
    let (stamp_w, stamp_h) = (schema.stamp_size_mm[0] * mm_to_pt,
                                   schema.stamp_size_mm[1] * mm_to_pt);
    let (off_x, off_y)   = (schema.position.offset_mm[0] * mm_to_pt,
                                   schema.position.offset_mm[1] * mm_to_pt);

    // 将 PDF 坐标映射到图像像素坐标
    let (img_w, img_h) = (img.width() as f32, img.height() as f32);
    let sx = img_w / page_w_pt;
    let sy = img_h / page_h_pt;

    // 检测区域 = 归档章位置（右上角）
    let roi_x     = (page_w_pt - off_x - stamp_w) * sx;
    let roi_y_img = off_y * sy;  // PDF y 原点在左下，图像在左上
    let roi_w     = stamp_w * sx;
    let roi_h     = stamp_h * sy;

    // 统计非白像素比例（RGB 均 > 240 视为白色）
    let mut non_white = 0u32;
    let mut total   = 0u32;
    for y in roi_y_img as u32..(roi_y_img + roi_h) as u32 {
        for x in roi_x as u32..(roi_x + roi_w) as u32 {
            if x < img_w as u32 && y < img_h as u32 {
                let p = img.get_pixel(x, y);
                if p[0] < 240 || p[1] < 240 || p[2] < 240 {
                    non_white += 1;
                }
                total += 1;
            }
        }
    }

    let ratio = if total > 0 {
        non_white as f32 / total as f32
    } else { 1.0 };

    // 非白色像素 < 10% → 判定为空白
    (ratio < 0.10, 1.0 - ratio)
}
```

## 十四、关键风险与对策

| 风险 | 概率 | 对策 |
|------|------|------|
| `pdfium-render` 中文字体嵌入失败 | 中 | 降级用 Python Sidecar（`reportlab` + `pypdf`） |
| 空白检测误判（扫描件噪点） | 高 | 仅作「提示」，最终决定权在用户；阈值 UI 可调 |
| 大 PDF（> 50MB）渲染慢 | 中 | 仅渲染第一页用于检测和预览；盖章时逐页处理不全加载 |
| 手动坐标跨文件复用准确性 | 中 | 按「同一目录 + 同一文件名模式」复用，不跨目录复用 |
| SQLite 并发写入 | 低 | `sqlx` 连接池 + 单线程批处理（按顺序处理，不并行写同一 DB）|
| 文件命名格式不统一 | 高 | 解析失败时红色标注，允许用户手动填写字段（预览表格直接编辑）|
| 正则生成不支持中文 | 高 | 使用 `[\w\u4e00-\u9fff]+` 匹配中英文字符；允许用户手动编辑正则 |
| 双面打印页码位置错误 | 中 | UI 明确提示「奇数页外侧=右侧，偶数页外侧=左侧」|
| 模板匹配误判 | 中 | 只作提示不自动切换；置信度阈值设为 80% |
| 自定义模板字段与归档章不同步 | 中 | 修改字段注册表时，自动检测归档章 cells 中是否有无效字段引用 |
| **破坏双层PDF结构** | **高** | **使用内容流追加（非替换）；绝不渲染页面为图像再写回；使用 lopdf 追加新内容流保留原图像层和文本层** |
| lopdf 追加内容流失败 | 中 | 添加详细错误日志；提供 Python sidecar 作为降级方案 |
| PDF注释被用户隐藏 | 低 | 使用内容流追加（非注释）；如用注释需明确提示用户 |

- 成功：记录到 SQLite `files` 表，`status = 'completed'`
- 失败：记录错误原因到 `error_msg` 字段
- 手动处理：记录手动坐标到 `manual_x_mm` / `manual_y_mm`

## 十六、最终执行流（完整流程概述）

```
1. 用户启动程序 → 选择/自动匹配档案类型模板
   ↓
2. （可选）切换到其他模板 或 新建自定义模板
   ↓
3. 选择根目录 → 程序扫描全部 PDF
   ↓
4. 根据当前模板的 B 区文件夹映射规则，自动划分批次
   ↓
5. 前端展示批次列表 + 文件解析预览表格
   ↓
6. 用户确认/调整 → 点击「开始处理」
   ↓
7. 逐文件处理：
   a. 根据当前模板 + 批次上下文 + C 区正则，提取字段值
   b. 按默认坐标计算归档章位置
   c. （可选）空白检测，如冲突则弹窗手动选址
   d. 绘制归档章到首页（lopdf，使用模板的 stamp_schema）
   e. 根据页码配置，遍历所有页面绘制页码
   f. 保存到 _stamped/ 目录
   g. 更新 SQLite 状态
   ↓
8. 处理完成 → 弹出完成报告（成功/失败统计）
```
